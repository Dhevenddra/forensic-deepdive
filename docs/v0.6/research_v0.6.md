## §1 — gRPC package-qualified keying (Step 5, DEC-068)
**Root cause of the 975 AMBIGUOUS cartesian joins:** protoc derives the generated Python module name
from the **.proto file path/name, NOT the `package` declaration.** Official Python Generated Code Guide
(protobuf.dev): "The Python code generated by the protocol buffer compiler is completely unaffected by
the package name… Python packages are identified by directory structure." And the filename transform:
"if the .proto file or its path contains any characters which cannot be used in Python module names (for
example, hyphens), they will be replaced with underscores. So, the file foo-bar.proto becomes the Python
file foo_bar_pb2.py." gRPC adds `_pb2_grpc.py` for the service stubs/servicers.

**Concrete divergence (the disambiguator):** `route_guide.proto` declares `package routeguide;` yet
generates `route_guide_pb2_grpc.py` — so the **module identity (`route_guide`) ≠ proto package
(`routeguide`)**. The grpc-examples monorepo reuses `Greeter`/`SayHello` across ~18 subdirs, all keyed
bare `grpc::Greeter/SayHello` → every client joins every servicer (cartesian, 975 AMBIGUOUS). Keying on
the **generated module** disambiguates: `helloworld_pb2_grpc::Greeter/SayHello` ≠
`route_guide_pb2_grpc::Greeter/SayHello`.

**Static recovery — both sides, AST-only (no `.proto`, no new dep):**
- **Servicer (provider):** `class Greeter(helloworld_pb2_grpc.GreeterServicer)` → base is
  `Attribute(value=Name('helloworld_pb2_grpc'), attr='GreeterServicer')`; module = `Name.id` resolved
  through an **import-alias table**; service = strip `Servicer`. Cross-check the registration call
  `helloworld_pb2_grpc.add_GreeterServicer_to_server(...)` (strip `add_`/`_to_server`).
- **Stub (consumer):** `stub = helloworld_pb2_grpc.GreeterStub(channel)` → module = `Name.id`, service =
  strip `Stub`; `stub.SayHello(...)` ties to the construction by intra-scope local-variable binding.
- **Import-alias table is mandatory** — all of `import helloworld_pb2_grpc` (flat, protoc default),
  `from . import helloworld_pb2_grpc` (relative/package), and `import users_pb2_grpc as users_service`
  (aliased) occur in the wild; projects also post-process protoc's flat imports.

**Key & confidence:** new key `grpc::<module>::<Service>/<Method>` (module = generated-module identity).
**EXTRACTED (confidence 1.0)** — every component is a deterministic protoc-emitted AST token; no
execution, no inference. **CAVEAT to record:** this is the *generated-module-identity* key, NOT the
on-the-wire path `/<proto_package>.<Service>/<Method>`. Equating module identity to the wire path needs
the `.proto`/descriptor → that variant is INFERRED and stays deferred behind `[proto]`. The recommended
v0.6 path keys on module identity (both sides reference the same generated module → they join correctly
without the `.proto`).

## §2 — ORM Django-vs-SQLAlchemy disambiguation (Step 1, DEC-064)
**The mis-tag:** Superset's `coremodel` (a non-Django class with a `Model` base) fell into the Django
branch because a bare `Model` base alone routed there → `orm='django'` on a SQLAlchemy table (1/55,
~2%; the `DbTable` + `PERSISTS_TO` edge stay correct — only the `orm` property is wrong).

**Require ≥1 Django-specific signal before the Django branch (strongest → weakest):**
1. `from django.db import models` / `from django.db.models import Model` import (strongest).
2. Base is the **qualified** `models.Model`, not a bare `Model`.
3. A nested `class Meta:` (Django idiom — `ordering`, `db_table`, `verbose_name`; abstract bases use
   `abstract = True`).
4. Field assignments `models.XField(...)` (`CharField`, `IntegerField`, `ForeignKey`).

**SQLAlchemy declarative counter-signals (fall-through target):** base from `declarative_base()` /
`DeclarativeBase`; a `__tablename__` attribute; `Column(...)` / `Mapped[...]` / `mapped_column(...)`
fields. **Rule:** Django branch fires on `(django.db.models import) OR (qualified models.Model base) OR
(nested Meta + a models.*Field)`; else fall through to SQLAlchemy/other. **EXTRACTED** (all syntactic).
Restores Superset to 55/55 correct ORM tags.

## §3 — RabbitMQ topic-exchange + binding-key topology (Step 4, DEC-067)
**Zero-match cause:** publishers `basic_publish(exchange='topic_logs', routing_key='kern.critical')`;
subscribers `queue_bind(exchange='topic_logs', queue=q, routing_key='kern.*')`. The two sides share no
literal — concrete dotted key vs wildcard pattern — so `queue::<literal>` keying yields 0 topic matches.

**AMQP 0-9-1 semantics (RabbitMQ Tutorial Five, official):** "`*` (star) can substitute for exactly
one word. `#` (hash) can substitute for zero or more words." Exchange types: **direct** = exact key;
**fanout** = ignore key; **topic** = wildcard pattern; **headers** = header args. A `#` binding behaves
like fanout; a wildcard-free topic behaves like direct.

**Mapping onto Endpoint/ROUTES_TO (keystone-preserving — `base.join` UNCHANGED):**
- **Key on the EXCHANGE, the shared literal:** `amqp::<exchange>` (e.g. `amqp::topic_logs`). Both
  `basic_publish(exchange=...)` and `queue_bind(exchange=...)` name the same exchange → `base.join`
  matches them by exact key, exactly as today.
- **Carry routing_key (publisher) and binding pattern (subscriber) as edge PROPERTIES.**
- **Refine the join by an AMQP wildcard matcher** (stdlib `re`: `*` → one word, `#` → zero-or-more,
  `.`-delimited) in a post-join reconcile step (the `reconcile_spec_backed` precedent, DEC-060 — contract
  layer, not `trace`/emit/serve): for each exchange-matched publisher↔subscriber candidate, test
  `routing_key` against `binding_pattern`:
  - exact literal match (direct exchange, or topic with no wildcards) → **EXTRACTED**;
  - wildcard match (`kern.*` ⊇ `kern.critical`) → **INFERRED**;
  - exchange matches but the pattern provably does NOT match the key → **DROP** (no false ROUTES_TO);
  - several subscribers match one publisher → **AMBIGUOUS** fan-out (emit all matching).
- **Direct/named-queue stays `queue::<name>`** (already works — the default exchange binds each queue by
  its own name as routing key). The new code path is topic/exchange only.

Takes rabbitmq-tutorials topic examples from **0 → >0** matched ROUTES_TO without touching `base.join`.

## §4 — JAX-RS sub-resource locators (Step 3, DEC-066)
**Zero-route cause:** the jersey `bookstore-webapp` is built on **sub-resource locators** — a
`@Path`-annotated method with **no HTTP-verb annotation** that **returns a resource object**. Jakarta
RESTful Web Services 3.0: a `@Path` method's *presence/absence of a request-method designator (`@GET`
etc.)* differentiates a sub-resource *method* from a sub-resource *locator*; a locator returns an object
that processes the remainder of the request.

```java
@Path("/")
public class ShoppingStore {
  @Path("/customers/{id}")
  public Customer getCustomer(@PathParam("id") int id) { return cust; }   // LOCATOR — no @GET
}
public class Customer {
  @GET public String get() {...}
  @Path("/address") public String getAddress() {...}
}
```

**Static resolution:** (1) detect `@Path` method with no verb → locator; (2) resolve the declared
return type (`Customer`, or `Class<ItemContentSingletonResource>` forms) to its class via the existing
cross-file resolver (`_resolve_name_to_files`, DEC-059); (3) recurse into that class's `@GET`/`@Path`
methods, concatenating the path prefix. **Confidence:** concrete annotated return type → **EXTRACTED**;
return type is `java.lang.Object` / an interface / an abstract type (spec permits, for runtime
polymorphic dispatch) → unresolvable statically → **AMBIGUOUS**. Takes jersey bookstore **0 → >0**.

## §5 — Django decoupled-route provider (Step 2, DEC-065)
Django routing is a spec-table in `urls.py` (route and handler in different files), unlike
decorator-on-handler. Shapes to parse:
- `path('owners/', views.owner_list)` → route + function-view symbol reference.
- `re_path(r'^archive/(?P<year>[0-9]{4})/$', views.archive)` → regex route.
- `path('my-url/', MyView.as_view())` → class-based view; resolve `MyView`.
- `path('blog/', include('blog.urls'))` / `include('app.urls')` → **prefix nesting**; recurse into the
  included module's `urlpatterns`, concatenate the prefix.
- DRF `router.register(r'authors', AuthorViewSet)` + `path('', include(router.urls))` → expand the
  ViewSet into the standard CRUD route set (list/create/retrieve/update/destroy) under the prefix.

**Static resolution:** a **provider extractor** (`contracts/http/providers/django.py`) emitting
`Endpoint(protocol='http')` + HANDLES across files, **reusing the existing key `http::<METHOD>::<path>`
(no new node type — keystone held).** Cross-file view resolution reuses `_resolve_name_to_files`
(DEC-059); `include()` prefixes recurse. **Confidence:** direct `path()`/`as_view()` references →
**EXTRACTED**; DRF default-router (SimpleRouter/DefaultRouter) CRUD expansion is deterministic →
**EXTRACTED-by-convention**, but a custom router or `@action` methods → **INFERRED**.

> **Strategic note:** GitNexus's open bug **#1183** is the same prefix-nesting failure on the *consumer*
> side (FastAPI `include_router(prefix=)` dropped → prefix-stripped `http::GET::/{param}` collisions →
> 0 cross-links). §5's recursion is the fix a 19k+-star competitor currently ships broken.

## §6 — Long-term memory landscape & positioning (Step 6, DEC-069)

### (A) Product-memory lane (iii) hardening — local-first, zero-LLM
- **Source of truth = markdown/JSONL files** (git-portable, human-diffable, agent-writable) — the
  existing `JsonlInsightStore` (DEC-019) stays authoritative.
- **Derived SQLite/FTS5 index, rebuildable** (delete it → rebuild from files) — **reuse the DEC-041
  FTS5/BM25 sidecar** infrastructure already in the project; no new dep (`sqlite3` ships FTS5).
- **BM25 lexical retrieval is the default** (no embeddings, no network — excellent for exact technical
  strings/symbols); the existing opt-in `[semantic]` ONNX path is an OPTIONAL hybrid layer fused via the
  existing **RRF (DEC-038)**.
- **Git-portability:** store insights on a dedicated git shadow-ref (precedent: Entire.io stashes
  session metadata on a hidden `entire/checkpoints/v1` branch).
- **Dedup** by content hash (SHA-256, the project's existing `ParseCache` discipline, DEC-036).
- **Decay/staleness:** a recency/importance score (optionally FSRS-style — Vestige/PowerMem precedent),
  kept entirely off the LLM path.

This pattern is what multiple 2026 local-memory tools converged on (memweave: "memories are .md files…
indexed into a local SQLite database… hybrid BM25 + semantic… the database is always a derived cache";
sqlite-memory, AIngram, OpenClaw — all SQLite + FTS5 + optional sqlite-vec). Everything here is stdlib or
already-DEC'd extras → **no floor breach.**

## §9 — Caveats
- gRPC module-identity key ≠ wire path; cross-language gRPC (Go/Java) is explicitly deferred (would need
  `[proto]` + per-language stub conventions).
- AMQP wildcard joins are inherently fan-out (one `*`/`#` matches many keys) → those ROUTES_TO edges are
  INFERRED/AMBIGUOUS, never EXTRACTED; provably-non-matching candidates are dropped, not emitted.
- JAX-RS locators returning `Object`/interface/abstract types are unresolvable statically (AMBIGUOUS).
- DRF custom routers / `@action` methods break the deterministic CRUD-expansion assumption (INFERRED).
- GitNexus (~19k–28.9k) and Graphify (~58k–63k) star counts vary by source/date — approximate, not
  load-bearing; the license + issue/bug facts are primary GitHub sources and reliable.
- Entire.io's "shared memory for agents" is forward-looking marketing (an adjacent session-provenance /
  git-shadow-branch tool, $60M seed Feb 2026), not a shipped code-memory product — a git-portability
  *reference*, not a competitor.
- Perf work is profiling-only; incremental/persistent graph update remains deferred to v1.0 (lane i).
  Full build ~20–40 min on 18–21k-symbol polyglot repos — the registry-dispatch fan-out + cross-stack
  join + DI/ORM passes are the standing profiling targets.
- Acceptance-repo scale figures are qualitative — confirm exact counts at fixture-selection time.
